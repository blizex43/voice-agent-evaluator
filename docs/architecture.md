# Architecture

This project is a Python voice-bot harness for evaluating a healthcare AI receptionist over Twilio. The entrypoint in `main.py` dispatches between configuration checks, local scenario simulation, webhook serving, and outbound calling. In the live flow, Twilio sends each user utterance to `POST /voice`, the server looks up the active conversation by `CallSid`, adds the latest turn to the conversation history, and asks the LLM for the next patient-side reply. The response is returned as TwiML so Twilio can speak it and collect the next user input until the call completes.

The main design choice was to keep telephony, conversation logic, scenarios, and artifact generation separate so the system stays easy to inspect and extend. Twilio-specific behavior lives in the caller and webhook modules, while the conversation layer focuses on maintaining context and requesting the next response. Scenario definitions and bug-detection rules live under `enums/`, which makes it straightforward to add new challenge cases without restructuring the call flow. The project also favors a simple, lightweight architecture over production-grade infrastructure: in-memory sessions are sufficient for local runs, and transcripts, reports, and recordings are persisted under `output/` for review and submission.
