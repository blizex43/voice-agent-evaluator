# Architecture

This project is a Python voice-bot harness that places outbound calls to the Pretty Good AI test line through Twilio, exposes a local FastAPI webhook through ngrok, and uses Groq as the LLM layer for generating realistic patient-side responses. `main.py` is the entrypoint: it can validate environment variables, run local simulations, start the webhook server, or place a real call. During a live call, Twilio posts speech turns to `POST /voice`; the webhook looks up the conversation by Twilio `CallSid`, passes the latest patient utterance plus recent history into the conversation engine, returns TwiML with `<Say>` and `<Gather>`, and repeats that loop until the call ends. Twilio status and recording callbacks are handled separately so transcripts, bug reports, recording metadata, and downloaded audio are written under `output/`.

The main design choice was to keep telephony, conversation logic, scenarios, and artifacts separate so the system is easy to test and change. Twilio-specific code lives in the caller/webhook modules, while `Conversation` only knows how to manage message history and ask the LLM for the next patient response. Scenario data and bug-detection rules live in `enums/` so new test cases can be added without changing the call flow. The system intentionally favors a simple, inspectable pipeline over production infrastructure: in-memory sessions are enough for local challenge runs, transcript/report files make results easy to submit, and the LLM provider is wrapped behind a small client so Groq can be swapped later if needed.
